# 常见陷阱与避坑

## 本章目标

- 识别使用 Claude Code 时的常见错误
- 掌握避免踩坑的方法
- 学会在问题发生后正确处理

---

## 沟通类陷阱

### 陷阱 1：需求描述太模糊

```
❌ 问题：
"帮我看看这个代码"

✅ 正确做法：
"帮我审查 src/user.py 中的登录函数，
重点检查：
1. 密码是否明文存储
2. 是否有 SQL 注入风险
3. 错误处理是否完善
"
```

**为什么踩坑**：模糊需求让 AI 只能猜测你想做什么，容易答非所问。

**避坑方法**：
- 用"我要你做 X，输出 Y 格式，关注 Z 点"的结构
- 提供具体的判断标准

---

### 陷阱 2：一次提太多要求

```
❌ 问题：
"帮我写一个用户注册功能，要包含邮箱验证、手机号验证、
密码强度检查、已经存在用户检测、还要有验证码、
注册成功发邮件、注册完自动登录..."

✅ 正确做法：
分步进行：
1. "先帮我写基本的注册表单 HTML"
2. "确认后，添加密码强度验证"
3. "然后添加邮箱格式验证"
4. ...
```

**为什么踩坑**：一次说太多，AI 可能遗漏、混淆或只做表面。

**避坑方法**：
- 每步聚焦一个功能
- 完成一步确认一步
- 用数字序号分隔不同要求

---

### 陷阱 3：没说清楚角色和场景

```
❌ 问题：
"怎么让文章更专业？"

✅ 正确做法：
"我要写给投资人看的商业计划书，
需要：
1. 用词正式、客观
2. 多用数据支撑
3. 避免口语化表达
参考风格：[附一段期望风格的示例]
"
```

**为什么踩坑**：AI 不了解你的背景和读者，无法给出合适的建议。

**避坑方法**：
- 明确目标读者
- 说明使用场景
- 提供风格示例

## 技术类陷阱

### 陷阱 4：直接运行 AI 写的代码

```
❌ 问题：
AI 写完代码，立刻运行
→ 报错 → 慌了 → 抱怨 AI 不行

✅ 正确做法：
1. 先让 AI 解释代码逻辑
2. 检查是否符合预期
3. 在测试环境运行
4. 确认没问题再正式使用
```

**为什么踩坑**：AI 写的代码可能有 bug、逻辑错误，甚至安全漏洞。

**避坑方法**：
- 始终先 review 代码
- 用测试环境验证
- 不在生产环境直接跑陌生代码

---

### 陷阱 5：不理解就复制

```
❌ 问题：
"这个错误是什么意思？" → AI 解释
"好的，谢谢" → 复制粘贴使用，不理解

→ 换个场景又出错
→ 不知道怎么改

✅ 正确做法：
1. 问"为什么这样改？"
2. 问"这个改动会影响其他地方吗？"
3. 问"下次遇到类似问题，怎么自己排查？"
```

**为什么踩坑**：不理解原理，换个场景就不会了。

**避坑方法**：
- 多问"为什么"
- 记录原理而非只是答案
- 让 AI 教你看懂代码

---

### 陷阱 6：忽视安全警告

```
❌ 问题：
AI 提示："这个操作有风险，确认要继续吗？"
用户："先做了再说"

→ 意外删除了重要文件
→ 执行了危险命令
→ 数据丢失

✅ 正确做法：
1. 认真阅读警告内容
2. 确认理解风险
3. 做好备份
4. 必要时拒绝执行
```

**为什么踩坑**：AI 的警告通常是经过判断的，忽视可能造成损失。

**避坑方法**：
- 看到警告先暂停
- 理解风险后再决定
- 重要操作前先备份

## 效率类陷阱

### 陷阱 7：上下文过长不清理

```
❌ 问题：
一个长项目聊了几十轮
→ AI 开始"遗忘"早期要求
→ 前后不一致
→ 用户觉得 AI 变笨了

✅ 正确做法：
1. 定期使用 /compact 压缩上下文
2. 长任务中间做总结确认
3. 阶段完成后开新对话
4. 用 CLAUDE.md 记住项目关键信息
```

**为什么踩坑**：上下文有长度限制，过长会导致 AI 遗忘重要信息。

**避坑方法**：
- 意识上下文长度
- 主动压缩和整理
- 不要在一个对话里做太多事

---

### 陷阱 8：用 Claude Code 做搜索引擎

```
❌ 问题：
"帮我查一下这个函数怎么用"
"这个错误是什么意思"
→ 每次都开新对话问

✅ 正确做法：
Claude Code 擅长：
- 需要理解和推理的问题
- 需要结合上下文的代码修改
- 需要多步骤的任务

简单查文档：
- 直接查官方文档
- 用搜索引擎

不要用牛刀杀鸡
```

**为什么踩坑**：简单问题用 AI 效率不高，浪费资源。

**避坑方法**：
- 简单语法查文档
- AI 做需要推理的任务

## 操作类陷阱

### 陷阱 9：没确认就继续

```
❌ 问题：
AI: "我已经修改了文件 A、B、C"
用户: "好的，继续"
→ 后来发现 B 改错了
→ 不知道是 AI 错还是需求变了

✅ 正确做法：
1. 每步让 AI 列出改了哪些文件
2. 主动确认改动是否正确
3. 必要时用 git diff 查看改动
4. 确认后再继续下一步
```

**为什么踩坑**：没确认就继续，出问题难以定位原因。

**避坑方法**：
- 养成确认的习惯
- 用 git 保存进度
- 不确认不继续

---

### 陷阱 10：忽略 AI 的建议

```
❌ 问题：
AI: "建议先备份数据库"
用户: "不用，太麻烦了"
→ 出问题了
→ 后悔

✅ 正确做法：
1. 认真听 AI 的建议
2. 评估风险
3. 重要操作听从建议
4. 除非有充分理由，不轻易跳过
```

**为什么踩坑**：AI 经验丰富，建议通常有道理。

**避坑方法**：
- 不轻易跳过警告和建议
- 问清楚为什么不听
- 风险操作必须有理由

## 问题发生后怎么办

### 快速恢复步骤

```
1. 停止：不要再继续执行任何操作
2. 评估：问题影响范围多大？
3. 回滚：能否恢复到之前状态？（git checkout）
4. 沟通：清楚描述问题
5. 解决：和 AI 一起排查
6. 预防：思考如何避免类似问题
```

### 和 AI 沟通问题的技巧

```
模板：
---
问题：描述发生的问题
预期：原本应该怎样
实际：实际发生了什么
影响：造成了什么后果
已尝试：已经试过的方法
---
```

```
使用 Claude Code 前检查：

□ 上下文是不是太长了？
□ 这个问题用 AI 合适吗？
□ AI 的警告我看了吗？
```
