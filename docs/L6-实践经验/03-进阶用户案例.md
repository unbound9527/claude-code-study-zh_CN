## 案例库

### 案例 1：后端工程师的日常

**场景**：重构一个 3000 行的遗留 Python 服务

**做法**：
```
1. 让 Claude Code 分析代码结构
   "分析这个 service.py，输出：
    - 主要函数列表
    - 函数依赖关系
    - 需要保留的公共 API"

2. 分模块重构
   每次只重构一个模块，每完成一个确认一个

3. 写测试
   "为这个函数生成单元测试，覆盖正常和异常情况"

4. 对比测试结果
   旧代码 vs 新代码，确保行为一致
```

**效果**：
- 3 周完成重构
- 零 regression bug
- 代码可读性从 C 级提升到 A 级

**心得**：
> "关键是分步和每步确认。Claude Code 帮了大忙，但必须盯着它走。"

### 案例 2：独立开发者的全栈开发

**场景**：一个人用 2 周开发 MVP

**做法**：
```
1. 产品设计（第 1 天）
   "我想要一个笔记分享应用，核心功能：...
   帮我生成产品设计文档"

2. 技术选型（第 1 天）
   "根据这个需求，推荐技术栈，解释为什么"

3. 后端开发（第 2-5 天）
   "用 FastAPI 实现以下 API...
    每个 API 写完后帮我测试"

4. 前端开发（第 6-10 天）
   "用 React + Tailwind 实现页面...
    每天完成 2-3 个页面"

5. 联调测试（第 11-12 天）

6. 文档和部署（第 13-14 天）
```

**效果**：
- 2 周完成平时可能需要 1 个月的功能
- 代码质量比自己写的还规范
- 有完整测试覆盖

**心得**：
> "以前觉得 AI 写代码不行，用了才发现——是你不会问。不是 AI 写不好，是你不会描述需求。"

### 案例 3：数据分析师的自动化

**场景**：每周自动生成数据分析报告

**做法**：
```
1. 创建标准化报告模板
   用 Claude Code 设计报告格式

2. 编写自动化脚本
   "写一个 Python 脚本：
    - 读取 Excel 数据
    - 生成统计图表
    - 输出 Markdown 报告"

3. 设置 Cron 定时任务
   每周一自动运行

4. Claude Code 配合
   运行后让 Claude Code 分析数据异常点
```

**效果**：
- 原本 4 小时的手工报表，变成 0
- 每周省下 4 小时
- 报告格式标准化，更专业

**心得**：
> "AI 自动化了操作，但思考还是得自己做。AI 是执行者的延伸，不是思考者的替代。"

### 案例 4：留学生的论文写作

**场景**：非英语母语留学生写英文论文

**做法**：
```
1. 先用中文写主要内容
   不管英文，先把想法写出来

2. 让 Claude Code 翻译
   "帮我把以下中文翻译成学术英文：
    - 保持专业语气
    - 使用学术常用表达
    - 标注我不确定的词汇"

3. 改写和润色
   "检查以下段落的语法和表达，
    改写成更地道的学术英语"

4. 降重处理
   "在不改变意思的前提下，
    改写以下段落以降低重复率"
```

**效果**：
- 写作效率提升 3 倍
- 语言质量明显提高
- 老师评价"英文进步很大"

**心得**：
> "不要让 AI 直接写，它不懂你的研究。用 AI 是翻译和润色，不是代写。"

### 案例 5：DevOps 的 IaC 管理

**场景**：管理 50+ 微服务的 Terraform 配置

**做法**：
```
1. 建立模块库
   "帮我创建标准化的 Terraform 模块：
    - network
    - ecs
    - rds
    每个模块有变量验证和输出定义"

2. 代码审查
   "审查这个 terraform 代码，
    找出配置错误和安全风险"

3. 文档生成
   "为这个 Terraform 配置生成 README，
    说明如何部署和使用"

4. 迁移脚本
   "帮我写一个脚本，
    将 AWS 区域 A 的资源迁移到区域 B"
```

**效果**：
- 新服务部署从 2 天缩短到 2 小时
- 配置错误率降低 80%
- 团队新人上手时间从 2 周变成 3 天

**心得**：
> "IaC 特别适合 AI。配置文件结构化，AI 容易理解，而且出错了可以随时回滚。"

### 案例 6：产品经理的需求分析

**场景**：把模糊需求转化成详细 PRD

**做法**：
```
1. 初始需求
   "用户需要一个搜索功能"

2. 追问细节
   "搜索的范围是什么？"
   "需要什么过滤条件？"
   "搜索结果怎么展示？"
   "性能要求是什么？"

3. 补充案例
   "举几个具体的搜索场景"

4. 生成 PRD
   "根据以上讨论，生成标准格式的 PRD"
```

**效果**：
- 需求文档质量大幅提升
- 和开发沟通更顺畅
- 减少开发过程中的需求变更

**心得**：
> "AI 的追问能力很强。用它来帮你思考，比你自己想更全面。"

## 常见问题

**Q: AI 写的代码能用吗？**
A: 能用，但要 review。不要直接跑没看过的代码。

**Q: AI 会取代程序员吗？**
A: 不会取代，但会取代"只会复制粘贴"的程序员。善用 AI 的程序员会更有价值。

**Q: 付费版值得吗？**
A: 如果每天用，值得。免费版适合学习和尝鲜。

## 本章小结

**L6 实践经验**是整本教程的"实战汇总"，把前面的知识转化为高效行动：

| 模块 | 核心要点 |
|------|---------|
| 高效模式 | 简单任务直接问，复杂任务配工具，长期任务用 Memory |
| 避坑指南 | 模糊需求是大坑，上下文要管理，备份意识不能少 |
| 进阶案例 | 看高手怎么用——不是 AI 不行，是你问的方式不对 |

**三条黄金法则**：
```
1. 需求说清楚 → AI 才能做对
2. 工具用到位 → AI 才能做快
3. 结果要验证 → AI 也会犯错
```

**心态调整**：
- ❌ "AI 怎么这么笨" → ✅ "我该怎么问才能让它做对"
- ❌ "AI 取代程序员" → ✅ "会用 AI 的程序员取代不会用的"
- ❌ "一次搞定" → ✅ "迭代优化才是常态"

**本章重点记住**：L6 是"知行合一"的检验。看十遍不如用一遍，真正的高手都是练出来的。
