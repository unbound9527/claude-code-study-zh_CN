# 权限系统：Tool 级别的三元拦截链

## 导言

大多数 CLI 工具的权限系统是"全开"或"全关"的二元选择。Claude Code 的权限系统是**Tool 级别的拦截链**：每一次工具调用都会经过 `canUseTool()` 的三元决策（allow/deny/ask），而这个决策可以被多个层面的规则影响——PermissionMode、alwaysAllowRules、Tool 自身的能力标志、以及运行时分类器。本文解析这个权限系统的全貌。


## 一、PermissionMode：四层总开关

### 1.1 四种模式

```typescript
export type PermissionMode =
  | 'acceptEdits'   // 自动接受所有编辑
  | 'bypassPermissions'  // 完全绕过权限检查
  | 'default'       // 用户确认（默认）
  | 'dontAsk'       // 自动拒绝所有询问
  | 'plan'          // Plan 模式（更严格的限制）
  | 'auto'          // 自动决策（实验性）
  | 'bubble'        // 冒泡模式

export const PERMISSION_MODES = [
  'acceptEdits', 'bypassPermissions', 'default',
  'dontAsk', 'plan',
  ...(feature('TRANSCRIPT_CLASSIFIER') ? ['auto']  []),
] as const
```

### 1.2 模式的影响

| 模式 | 效果 |
|------|------|
| `acceptEdits` | 所有编辑类操作自动允许 |
| `bypassPermissions` | 完全跳过权限检查（慎用） |
| `default` | 用户逐个确认 |
| `dontAsk` | 所有请求自动拒绝 |
| `plan` | Plan 模式，更严格的限制 |
| `auto` | 由分类器自动决策（实验性） |


## 二、三元权限决策

### 2.1 决策类型

```typescript
// types/permissions.ts
export type PermissionBehavior = 'allow' | 'deny' | 'ask'

// 决策结果
type PermissionDecision<T> =
  | { behavior 'allow'; updatedInput? T; decisionReason? DecisionReason }
  | { behavior 'deny'; reason string; decisionReason? DecisionReason }
  | { behavior 'ask'; decisionReason? DecisionReason }
```

### 2.2 `canUseTool()` 决策链

`src/hooks/useCanUseTool.tsx` 是权限检查的核心：

```typescript
// useCanUseTool.tsx
const decisionPromise = forceDecision !== undefined
  ? Promise.resolve(forceDecision)  // 强制决策（测试用）
  : hasPermissionsToUseTool(tool, input, toolUseContext, assistantMessage, toolUseID)
```

`hasPermissionsToUseTool()` 是真正的决策函数：

```typescript
// permissions/permissions.ts
export function hasPermissionsToUseTool(
  tool: Tool,
  input: unknown,
  toolUseContext: ToolUseContext,
  assistantMessage: AssistantMessage,
  toolUseID: string,
): Promise<PermissionDecision>
```


## 三、三层规则系统

### 3.1 规则来源

Claude Code 的权限规则来自多个源（types/permissions.ts）：

```typescript
type PermissionRuleSource =
  | 'userSettings'    // ~/.claude/settings.json
  | 'projectSettings' // .claude/settings.json
  | 'localSettings'   // .claude.local.json
  | 'flagSettings'     // 命令行标志
  | 'policySettings'   // /etc/claude-code/ 策略
  | 'cliArg'           // --permission-mode
  | 'command'          // slash 命令
  | 'session'         // 会话级别
```

### 3.2 alwaysAllowRules

每个 PermissionMode 关联一个 `alwaysAllowRules`，用于自动允许特定工具：

```typescript
// 工具过滤在 QueryEngine.ts
setAppState(prev => ({
  ...prev,
  toolPermissionContext: {
    ...prev.toolPermissionContext,
    alwaysAllowRules: {
      ...prev.toolPermissionContext.alwaysAllowRules,
      command: allowedTools,  // 来自 slash 命令的工具自动允许
    },
  },
}))
```

### 3.3 Tool 级别的 `checkPermissions()`

每个 Tool 可以定义自己的权限检查逻辑（Tool.ts）：

```typescript
checkPermissions(
  input: z.infer<Input>,
  context: ToolUseContext,
): Promise<PermissionResult>
```

例如 `BashTool` 会检查命令是否涉及 `sudo`、`rm -rf` 等高危操作。


## 四、分类器决策（实验性）

### 4.1 BashClassifier

当 `feature('BASH_CLASSIFIER')` 启用时，`BashTool` 会使用分类器来判断命令是否安全：

```typescript
// BashTool/bashPermissions.ts
export function startSpeculativeClassifierCheck(
  tool: Tool,
  input: unknown,
  toolUseContext: ToolUseContext,
): Promise<PendingClassifierCheck | null>
```

这个分类器可以：
- 分析 bash 命令的语义
- 判断是否为常见的开发操作
- 自动允许低风险命令，拒绝高风险命令

### 4.2 Auto Mode

当 `feature('TRANSCRIPT_CLASSIFIER')` 启用时，存在 `auto` 权限模式：

```typescript
if (feature('TRANSCRIPT_CLASSIFIER') && result.decisionReason?.type === 'classifier' && result.decisionReason.classifier === 'auto-mode') {
  setYoloClassifierApproval(toolUseID, result.decisionReason.reason)
}
```


## 五、InteractiveHandler：权限对话框

### 5.1 交互式权限处理

当决策为 `ask` 时，`handleInteractivePermission()` 显示权限对话框：

```typescript
// useCanUseTool.tsx
case 'ask' {
  if (appState.toolPermissionContext.awaitAutomatedChecksBeforeDialog) {
    const coordinatorDecision = await handleCoordinatorPermission({ ... })
    // ...
  }
  // 显示交互式权限对话框
  await handleInteractivePermission({ ... })
}
```

### 5.2 权限对话框的钩子

权限对话框提供了多个钩子：
- `PreToolUse` 钩子：在工具执行前运行
- `PostToolUse` 钩子：在工具执行后运行

## 七、权限与 Tool 系统的交互

### 7.1 filterToolsByDenyRules

`tools.ts` 有一个预过滤步骤：

```typescript
export function filterToolsByDenyRules<T>(
  tools: readonly T[],
  permissionContext: ToolPermissionContext,
): T[] {
  return tools.filter(tool => !getDenyRuleForTool(permissionContext, tool))
}
```

这把权限检查提前到**API 调用之前**——如果一个工具被 deny 了，它根本不会出现在发送给模型的 tool schema 中。

### 7.2 拒绝 vs 不发送

这是两个不同层级的限制：

| 层级 | 机制 | 效果 |
|------|------|------|
| **不发送** | `filterToolsByDenyRules` | 工具 Schema 不出现在 prompt 中 |
| **拒绝执行** | `canUseTool()` 返回 deny | 工具在 prompt 中，但执行时被拦截 |

"不发送"是更彻底的限制——模型甚至不知道有这个工具。


## 总结

Claude Code 的权限系统是一个**分层、多源、可组合的拦截链**：

1. **PermissionMode 是总开关**：决定整体权限策略
2. **alwaysAllowRules 是白名单**：特定工具自动允许
3. **Tool 级 checkPermissions() 是自定义策略**：每个工具可以有自己的判断逻辑
4. **分类器是自动决策**：实验性的自动权限决策能力
5. **InteractiveHandler 是用户交互层**：需要用户做决策时弹出对话框

这个设计的精妙之处在于：规则是可组合的，多个规则可以同时影响同一个决策，而不是简单的覆盖。
